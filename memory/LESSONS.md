# LESSONS — durable rules the desk has learned (consult before trading)

Seeded with priors from the strategy. Add to this whenever a trade teaches something. Keep each lesson as a rule that changes future behavior.

## Starting priors
- **The core is the edge.** 70% in SPY means we start each week nearly matching the index. Don't jeopardize that reliability chasing satellite thrills.
- **Cut fast, let winners run.** Trailing stops exist so losers are small and winners are large. Never widen a stop; never average down.
- **A thesis or no trade.** If you can't write one falsifiable line for why, skip it. Boredom is not a signal.
- **Respect the caps.** The 5%/position, 3-per-week, and 3% daily loss cap are the difference between a strategy and gambling. The script enforces them — treat a rejection as wisdom, not an obstacle.
- **Beating SPY is hard.** Most active trading underperforms. Assume the burden of proof is on every satellite trade to justify its risk over just holding the index.
- **Don't revenge-trade.** After a stop-out, the next trade must clear the same bar as any other. Emotion is the enemy.

## Learned lessons
- **Never wrap an order-submitting command in a shell `||` fallback.** On 2026-07-08 a `python3 scripts/alpaca.py buy XLF ... || python scripts/alpaca.py buy XLF ...` pattern (used to try `python3` then fall back to `python`) caused a duplicate live buy: the first invocation submitted the market order successfully but exited non-zero when the trailing-stop attach step hit a transient race ("cannot open a short sell while a long buy order is open"), so the `||` triggered the second invocation, which submitted a second full buy order. Result: a satellite position at ~7.9% of equity (over the 5% cap) and a double-counted weekly slot, both caught and corrected after the fact only by manually checking `positions`/`orders` against intent. Fix: run buy/sell/stop commands as a single invocation (pick one interpreter and commit to it, e.g. probe with `python3 --version` once, or just run `python3 ...` and report failure rather than silently retrying with a fallback). If a command exits non-zero after "submitted" appears in its output, check live `positions`/`orders` before doing anything else — do not assume it's safe to retry.
- **The same 403 "cannot open a short sell while a long buy order is open" error also fires for a legitimate reason: a slow paper-market fill outlasting the script's 15s wait.** On 2026-07-09 the `buy XLK` command hit this exact error, but `orders` showed the buy order genuinely still `new`/`partially_filled` (not a duplicate) — it finished filling ~30s after submission. Confirmed this was not a repeat of the 2026-07-08 duplicate-order bug (`counters.json` stayed correctly at +1). Fix in this case: don't retry the buy; poll `orders`/`positions` until the buy shows fully filled (no open buy order left, position qty matches intended), then attach the trailing stop manually with `scripts/alpaca.py stop SYMBOL --trail-percent N`.
- **An overbought RS flag (RS>70) that appears shortly after a satellite entry is a cue to tighten the stop, not just a note to "watch."** XLV (entered 2026-07-07) was flagged RS>70/overbought on 2026-07-08, one day after entry, and then faded to -2.4% by 2026-07-10 without ever tightening its stop off the default 10%. The strategy already calls for tightening trails on "extended winners at midday" — the same logic applies when a fresh entry gets flagged overbought and then stalls: don't wait passively for the default 10% trail to be tested, proactively tighten (e.g. to 6-8%) per `scripts/alpaca.py stop SYMBOL --trail-percent N` the next time this pattern repeats.
- **`scripts/alpaca.py stop SYMBOL --trail-percent N` cannot retighten an *existing* trailing stop — it will be rejected.** Tried on 2026-07-14 to tighten XLV's stop from 10% to 7% after a sharp -3.9% single-day move; Alpaca returned a 403 (`insufficient qty available... held_for_orders`) because the original stop order already holds all the shares as sell-side inventory, and there is no cancel/replace subcommand in `scripts/alpaca.py` to free them first. Do **not** work around this by calling the Alpaca REST API directly (cancelling an order that way is exactly the kind of raw-call workaround the operating instructions forbid) — instead leave the existing stop live (it's still a valid safety net) and flag the gap in `memory/STATE.md`/here. This needs a proper fix: add a `cancel`/`replace-stop` subcommand to `scripts/alpaca.py` (cancel the old trailing-stop order by ID, then submit the new one) so future routines can actually retighten stops on flagged laggards instead of only being able to attach a stop once.
- **The same "no cancel/replace" gap also blocks a full discretionary exit, not just stop-retightening.** On 2026-07-17, XLK's thesis broke cleanly (price below its tracked 50-day MA for multiple sessions, confirmed worsening chip-sector rout) and the strategy's "midday discretionary cut" rule called for closing it early. `scripts/alpaca.py sell XLK --all` was rejected with the identical 403 `held_for_orders` error — a resting trailing-stop order holds 100% of the shares, so **no sell of any kind (discretionary or otherwise) is possible on a satellite position while its trailing stop is live**, only the stop itself can ever close the position until this is fixed. Left XLK open with its original stop, did not work around it via raw API. This is a second, independent confirmation that `scripts/alpaca.py` needs a cancel-order subcommand — without it, the desk's discretionary-exit discipline (a core part of the strategy) is currently non-functional for any position past its initial entry.
- **This gap is now confirmed structural (three occurrences: XLV retighten 7/14, XLK exit attempt 7/17 pre-open flag, XLK exit attempt 7/17 open), so the desk must change behavior around it, not just keep flagging it.** Until `scripts/alpaca.py` gets a cancel/replace-order subcommand: (1) treat every new satellite entry as effectively **un-cuttable except by its own trailing stop** once the stop is attached — factor that into the entry bar (size and conviction) since "we can exit early if the thesis breaks" is not currently true in practice; (2) for a position whose thesis breaks and can't be sold, consider whether tightening is somehow achievable, and if truly nothing is actionable, explicitly say so in `STATE.md` rather than repeatedly re-attempting the same rejected sell each routine; (3) escalate this as the top engineering priority in every weekly review until it's fixed — it sits directly on the critical path of the strategy's exit discipline.
